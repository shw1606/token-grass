# TokenGrass — 실행 로드맵 & 빌드인퍼블릭 콘텐츠 계획

> **포지셔닝:** 돈 버는 제품이 아니라 **빌더 정체성 바이럴 콘텐츠**. 제품의 바이럴보다 "만든 스토리"의 바이럴을 노린다.
> **한 줄 차별화:** *Mac 없이, iPhone 홈스크린에, 잔디밭만. 무료 · 오픈소스 · 페이월 제로.*
> **상태:** Draft v0.2 (경쟁 분석 반영)

---

## 0.5 경쟁 분석 요약 (실측 완료)

가장 가까운 경쟁자 `Usage for Claude`를 직접 설치해 확인:
- 잔디밭 기능은 **있으나 앱 내부 전용** — 홈스크린 위젯엔 잔디가 없다(게이지만).
- 데이터 수집에 **Mac 컴패니언 필수**.
- 풀 히스토리/고급 기능 **유료(IAP)**.

→ 비어 있는 정확한 교집합 = **위젯 잔디 ∩ 폰 단독 ∩ 무료**. 우리는 여기만 친다.
→ 리스크: 경쟁 개발자가 위젯 잔디를 추가할 수 있음. **속도가 해자.** 자세한 표는 DESIGN.md §1.3.

## 1. 단계별 로드맵

### Phase 0 — Spike (1–2일) ★먼저
> 여기서 빨간불 나오면 전체 전략 재검토. 코드 짜기 전에 무조건.
- [x] **경쟁 검증 (완료):** `Usage for Claude` 직접 설치 → 잔디는 앱 내부 전용·유료, 위젯엔 없음, Mac 필수 확인. 차별점(위젯 잔디 ∩ 폰 단독 ∩ 무료) 유효 판정.
- [ ] **R1:** 실제 Claude Code 토큰으로 usage 엔드포인트 호출 → 응답 스키마 캡처/고정
- [ ] **R2:** 폰/외부 환경에서 refresh 엔드포인트 호출 가능한지 확인 (Cloudflare 차단 여부)
- [ ] 결과로 토큰 전략 확정: `refresh 지원` vs `setup-token 장기토큰 전용`
- **산출물:** 짧은 기술 메모 → 그대로 첫 Threads 빌드로그 소재

## 2. 콘텐츠 계획 (채널 전략 연계)

### 2.1 채널 역할
- **Threads (@yulebuilds):** 진행형 빌드로그. 짧고 자주. 스샷·짧은 영상·삽질 기록.
- **LinkedIn:** 메타 레벨 에세이. 마일스톤마다 1편. "왜/무엇을 배웠나".
- **블로그 (seo-hnoo.me):** 캐노니컬 회고 1편 (출시 후). 두 채널의 허브.

### 2.2 빌드로그 비트 (Threads — 시간순)
| 시점 | 훅 | 형식 |
|---|---|---|
| Phase 0 | "Claude 잔디밭 앱 다 깔아봤다. 근데 홈화면 위젯에 잔디 띄우는 건 아무도 안 함 (+게다가 유료). 내가 만든다" | 텍스트+경쟁 비교 — 차별점 선언 |
| Phase 1 | "가짜 데이터지만... 홈화면에 떴다 🌱" (홈스크린 스샷) | 이미지 — **최대 떡밥** |
| Phase 1.5 | WidgetKit 30MB 메모리 삽질기 | 텍스트 (개발자 공감대) |
| Phase 2 | "실데이터 연결. Mac 없이 폰만으로 내 진짜 잔디다" | 이미지/짧은 영상 — 폰 단독 강조 |
| Phase 2.5 | "토큰 어떻게 안전하게 다뤘나 — Keychain only" | 보안 설계 (신뢰 빌드) |
| Phase 4 | "무료·오픈소스로 풉니다" + 레포 링크 | 링크+GIF |

### 2.3 LinkedIn 에세이 후보 (메타 레벨)
- "비공식 API에 의존하는 사이드 프로젝트, 어디까지 가도 되나 — 플랫폼 리스크를 안고 빌드하기"
- "1인 빌더가 위젯 하나로 빌드인퍼블릭 한 기록"
- "기술 moat 없는 프로젝트를 '콘텐츠 자산'으로 전환하는 법"

### 2.4 바이럴 트리거 설계 (제품 안에 심기)
- 위젯 자체가 스샷 공유 욕구를 부름(=GitHub 잔디 심리). 출시 후 v1.1에 "잔디 공유 이미지 export" 추가 검토.
- README 상단에 데모 GIF — 레포 그 자체가 콘텐츠.
- "당신의 잔디를 보여주세요" 식 가벼운 UGC 유도(Threads).
